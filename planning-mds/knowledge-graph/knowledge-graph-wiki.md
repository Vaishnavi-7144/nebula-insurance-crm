# Nebula Knowledge Graph Wiki

This is the human and LLM-readable map for Nebula's knowledge graph. It wraps
the YAML data in this directory into a navigable memory wiki without replacing
the YAML or the query tools.

The graph's job is to make product memory compound. Each feature run leaves
behind stable links from product intent to architecture, contracts, policies,
source files, tests, and symbols, so the next planning or implementation task
starts from a routed slice instead of broad repository search.

## Source Of Truth

The knowledge graph is a retrieval and routing index. It is not the authority
when it conflicts with raw artifacts.

Authority order:

1. Target feature folder and `feature-assembly-plan.md` for feature-local
   implementation intent.
2. `planning-mds/architecture/decisions/**` for architecture decisions.
3. `planning-mds/api/*.yaml` and `planning-mds/schemas/*.json` for interface
   and payload contracts.
4. `planning-mds/architecture/data-model.md` and
   `planning-mds/domain/glossary.md` for domain definitions.
5. `planning-mds/knowledge-graph/*.yaml` for compressed retrieval and routing.

If the graph and raw artifacts disagree, fix the raw artifact first when it is
wrong, then repair the graph in the same change set.

## Mental Model

```text
solution-ontology.yaml
  defines vocabulary, edges, ownership, and retrieval rules

canonical-nodes.yaml
  declares shared solution semantics

feature-mappings.yaml
  links feature/story scope to canonical semantics

code-index.yaml
  binds canonical semantics to implementation paths

symbol-index.yaml
  extracts symbols and call relationships from bound paths

coverage-report.yaml
  reports freshness, coverage, exclusions, and graph health

lookup.py / hint.py / blast.py / risk.py
  materialize the slice an agent or human should actually read
```

The graph is intentionally split this way. Human-curated structure stays small
and reviewable. Generated layers can be large because they are rebuilt from
source and validated.

## What Each Layer Answers

| Question | Start Here | Then Use |
|----------|------------|----------|
| What node and edge types exist? | [solution-ontology.yaml](solution-ontology.yaml) | `python3 scripts/kg/validate.py` |
| What canonical concepts does Nebula know about? | [canonical-nodes.yaml](canonical-nodes.yaml) | `python3 scripts/kg/lookup.py <node-id>` |
| Which features touch a concept? | [feature-mappings.yaml](feature-mappings.yaml) | `python3 scripts/kg/blast.py <node-id>` |
| What does a feature affect? | [feature-mappings.yaml](feature-mappings.yaml) | `python3 scripts/kg/lookup.py FNNNN` |
| Which source files implement a concept? | [code-index.yaml](code-index.yaml) | `python3 scripts/kg/lookup.py --file <path>` |
| Which symbols are inside bound source? | [symbol-index.yaml](symbol-index.yaml) | `python3 scripts/kg/lookup.py --symbol <name>` |
| Is graph coverage fresh? | [coverage-report.yaml](coverage-report.yaml) | `python3 scripts/kg/validate.py` |
| Did a code change exceed the planned blast radius? | [code-index.yaml](code-index.yaml) and [symbol-index.yaml](symbol-index.yaml) | `python3 scripts/kg/diff-impact.py <range>` |
| Is a node high-risk? | [coverage-report.yaml](coverage-report.yaml) and generated graph signals | `python3 scripts/kg/risk.py <node-id>` |

## Canonical Node Families

`canonical-nodes.yaml` groups shared solution memory by node family:

- `entities` - domain and append-only business records.
- `workflows` - cross-feature lifecycle or operating flows.
- `capabilities` - shared runtime, UI, operational, or platform behavior.
- `glossary_terms` - domain vocabulary anchors.
- `endpoints` - API surfaces used for routing and tests.
- `ui_routes` - frontend page and route surfaces.
- `events` - audit, timeline, workflow, or operational events.
- `config_keys` - runtime and deployment configuration surfaces.
- `migrations` - schema-shaping rollout artifacts.
- `roles` - identity and authorization roles.
- `policy_rules` - Casbin resource/action rules.
- `evidence` - validation and signoff artifacts.
- `adrs` - architecture decision records.
- `api_contracts` - OpenAPI or API contract nodes.
- `schemas` - JSON Schema and payload contract nodes.

Add a canonical node only when the concept is shared, stable, and useful for
future routing. Feature-local details should remain in the feature folder.

## Feature And Story Memory

`feature-mappings.yaml` maps delivery scope into the canonical layer.

Feature entries typically answer:

- What product scope is being described?
- Which canonical entities, capabilities, workflows, endpoints, routes, events,
  config keys, migrations, roles, and policies does it affect?
- Which ADRs govern it?
- Which schemas and API contracts does it use?
- Which earlier features or stories does it depend on?
- Is the feature active, complete, archived, draft, or intentionally excluded?

Story entries do the same at a smaller implementation-slice level. They are
especially useful when a feature touches multiple workflows or surfaces.

Excluded features are explicit. An excluded feature is not forgotten; it is
marked as intentionally unmapped so validators can distinguish planned gaps
from accidental coverage loss.

## Code And Symbol Memory

`code-index.yaml` is the semantic binding between canonical nodes and source
paths. It should bind cohesive surfaces by directory glob when possible, not
duplicate every file when a stable folder captures the capability.

`symbol-index.yaml` is generated from those bound paths. It exists so agents
can ask targeted questions:

- Where is this method/class/function defined?
- Which tests or callers reference it?
- Which implementation satisfies this interface member?
- What line range should an impact tool attribute to a change?
- Is there duplicate or near-duplicate surface before adding a new one?

The symbol layer intentionally does not scan the whole repository. Its scope is
the curated source boundary in `code-index.yaml`.

## Lifecycle

The graph is maintained as part of normal feature work:

| Trigger | Owner | Graph Update |
|---------|-------|--------------|
| New feature planned | Product Manager | Seed a minimal feature entry in [feature-mappings.yaml](feature-mappings.yaml). |
| Architecture completed | Architect | Complete feature/story mappings and add canonical nodes when shared semantics were introduced. |
| New source surface shipped | Architect | Add or confirm [code-index.yaml](code-index.yaml) bindings. |
| Code changed inside bound paths | Generated | Regenerate [symbol-index.yaml](symbol-index.yaml). |
| Feature completed and archived | Product Manager | Update feature path/status in [feature-mappings.yaml](feature-mappings.yaml), then refresh [coverage-report.yaml](coverage-report.yaml). |
| Policy changed | Architect | Keep `policy.csv` and `policy_rule` canonical nodes aligned. |
| Source moved or renamed | Architect | Update code bindings, then regenerate symbols and validate. |

The important feature-action gates are:

- Phase A: PM seeds mappings for touched features.
- Phase B: Architect finalizes canonical semantics.
- G7: Architect reconciles the graph against as-built source.
- G8: PM closeout updates lifecycle path/status and refreshes path-sensitive
  coverage after archive moves.

## Query Recipes

Run these from the product root.

```bash
# Validate graph integrity and freshness.
python3 scripts/kg/validate.py

# Refresh the generated coverage report after feature path/status changes.
python3 scripts/kg/validate.py --write-coverage-report

# Regenerate symbols after edits in bound source files.
python3 scripts/kg/validate.py --regenerate-symbols --check-symbols

# Check policy and memory-path drift.
python3 scripts/kg/validate.py --check-drift

# Get the graph slice for a feature.
python3 scripts/kg/lookup.py F0017

# Get a compact hint for a file before searching.
python3 scripts/kg/hint.py engine/src/Nebula.Api/Endpoints/SubmissionEndpoints.cs

# See the blast radius for a canonical node.
python3 scripts/kg/blast.py entity:submission

# Find all symbols by source name.
python3 scripts/kg/lookup.py --symbol TransitionAsync

# Check whether a proposed symbol/service name already exists.
python3 scripts/kg/lookup.py --defines TransitionAsync

# Score a canonical node before a risky edit.
python3 scripts/kg/risk.py entity:policy
```

## Edit Rules

Use this checklist before changing graph files:

- Edit [solution-ontology.yaml](solution-ontology.yaml) only when the
  vocabulary itself changes.
- Edit [canonical-nodes.yaml](canonical-nodes.yaml) when a shared concept,
  rationale, source doc, policy, endpoint, schema, route, workflow, or
  capability changes.
- Edit [feature-mappings.yaml](feature-mappings.yaml) when feature/story scope,
  status, path, dependencies, governance, schemas, policies, or affected nodes
  change.
- Edit [code-index.yaml](code-index.yaml) when implementation surfaces move or
  new source folders/files become canonical bindings.
- Do not hand-edit [symbol-index.yaml](symbol-index.yaml),
  [coverage-report.yaml](coverage-report.yaml), [decisions-index.yaml](decisions-index.yaml),
  or [unbound-but-referenced.yaml](unbound-but-referenced.yaml). Regenerate them.
- When in doubt, raw artifacts win. The graph should route agents toward the
  truth, not become a second source of truth.

## Failure Modes

| Symptom | Likely Cause | Response |
|---------|--------------|----------|
| `lookup.py FNNNN` returns only the target | Feature mapping is missing or intentionally excluded | Read feature files directly, then backfill a minimal mapping if the scope is active. |
| `validate.py` reports unknown refs | Mapping points at a missing canonical ID | Add the canonical node only if it is real and shared; otherwise correct or remove the ref. |
| A glob matches zero files | Source moved or binding is stale | Repair [code-index.yaml](code-index.yaml), regenerate symbols, and validate. |
| Coverage report is stale | Path/status/binding changed after last generated report | Run `python3 scripts/kg/validate.py --write-coverage-report`. |
| Symbol warnings reference missing symbols | Generated symbol/call data no longer matches the current bound set | Regenerate symbols and inspect unresolved edges if warnings remain. |
| A PM closeout finds missing semantic bindings | G7 reconciliation missed a source surface | Route back to Architect for a G7 delta; closeout should verify, not author semantic bindings. |
